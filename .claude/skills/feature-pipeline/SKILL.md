# feature-pipeline

Garminアプリに新しい機能を追加するための、4エージェントによる自動パイプラインを実行する。

引数（任意）: ランナー（ユーザー）からの相談内容や悩み。例: `最近テンポ走がつらい`。引数が空の場合は「次に追加すべき機能を一般的なトレーニング科学の観点から提案してほしい」という依頼として扱う。

## 手順

各ステップは前段の出力に依存するため、**Agent呼び出しは並列にせず必ず逐次実行**すること。

1. **現状把握**: `garmin/PLAN.md` を読み、必要であれば `garmin/FEATURE_LOG.md`（存在する場合）も読んで、現在アプリが利用者に提供している機能を利用者目線の箇条書きで要約する（DBカラム名やAPIパスなどの技術的詳細は含めない。「日付・種別・距離・時間・ペース・平均心拍・カロリーを一覧表示できる」といった粒度）。

2. **ナレッジ読み込み**: `garmin/.claude/knowledge/running-science/` 配下の全ファイル（`00_README.md`含む）を`Read`で読み込む。

3. **コーチ呼び出し**: `Agent` toolで `subagent_type: garmin-coach` を呼び出す。プロンプトには (1)手順1で作った利用者向け機能一覧、(2)ユーザーからの相談内容（空なら上記のデフォルト依頼文）、(3)手順2で読み込んだランニング科学ナレッジの全文を渡す。戻り値として機能提案（提案機能/根拠/効果/価値）を得る。

4. **情シス呼び出し**: `Agent` toolで `subagent_type: garmin-it-liaison` を呼び出す。プロンプトには (1)手順3のコーチの提案全文、(2)手順1の利用者向け機能一覧を渡す。戻り値として確認結果・ユーザーストーリー形式の要件・懸念点を得る。

5. **コンサル呼び出し**: `Agent` toolで `subagent_type: garmin-consultant` を呼び出す。プロンプトには手順4の要件全文を渡す（コンサル自身が `garmin/PLAN.md` や既存コードを `Read`/`Grep`/`Glob` で確認するため、パスの案内だけで十分）。戻り値として設計概要・影響範囲(DB/API/UI)・実装時の注意点を得る。

6. **ログ記録**: ここまでの結果を `garmin/FEATURE_LOG.md` に1エントリとして追記する（ファイルが無ければ作成する）。フォーマットは下記の「FEATURE_LOG.mdのエントリ形式」を参照。この書き込みは呼び出し元（このスキルを実行しているセッション自身）が行うこと。garmin-coach/garmin-it-liaison/garmin-consultantはファイル書き込み権限を持たないため、代わりに書き込む。

7. **コーディング呼び出し**: `Agent` toolで `subagent_type: garmin-coder` を呼び出す。プロンプトには手順5のコンサルの設計全文を渡す。実装を任せ、変更ファイル一覧と実装サマリを得る。

8. **ログ追記・報告**: 手順7の実装結果を手順6で作成したエントリに追記する。最後にユーザーへ、今回追加した機能・変更ファイル・動作確認が必要な点（実際のGarminアカウントでの確認など）を簡潔に報告する。

## FEATURE_LOG.mdのエントリ形式

```markdown
## YYYY-MM-DD: <提案機能名>

**ランナーからの相談**: <引数の内容、なければ「(なし・コーチによる自発提案)」>

**コーチ提案**:
<手順3の出力>

**情シス確認要件**:
<手順4の出力>

**コンサル設計**:
<手順5の出力>

**コーディング実装結果**:
<手順7の出力>
```

## 注意点

- コーチ・情シスの出力に技術的な内容（ファイルパス、DBカラム名、API名など）が混入していないか目を通し、混入していれば役割逸脱として扱い、該当エージェントを役割に沿った出力になるよう再度呼び出すことを検討する
- コンサル・コーディング以外のエージェントにファイル書き込みをさせないこと
- 1回の実行で提案する機能は1件に絞る（複数機能を一度に詰め込まない）
