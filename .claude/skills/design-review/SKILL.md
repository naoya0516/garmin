# design-review

`garmin/docs/` の基本設計書のうち、利用者・機能に関わる章をコーチ→情シスの順でレビューさせる。DB設計・API設計等の技術章はコーチ・情シスの権限方針（コード非可視）と整合しないため対象外とする。

引数: なし。

## 手順

各ステップは前段の出力に依存するため、**Agent呼び出しは並列にせず必ず逐次実行**すること。

1. **設計書読み込み**: `garmin/docs/01_システム概要.md` `garmin/docs/02_機能一覧.md` `garmin/docs/03_画面設計.md` の3章を`Read`する。`04_DB設計.md`以降の技術章は読まない。

2. **ナレッジ読み込み**: `garmin/.claude/knowledge/running-science/` 配下の全ファイルを`Read`する。

3. **コーチ呼び出し**: `Agent` toolで `subagent_type: garmin-coach` を呼び出す。プロンプトには (1)手順1の3章全文、(2)手順2のナレッジ全文、(3)「これはGarminアプリの基本設計書（利用者・機能に関する章）です。トレーニング科学の観点から、良い点・懸念点・追加で検討すべき機能があればレビューしてください」という指示を渡す。戻り値としてレビュー内容を得る。

4. **情シス呼び出し**: `Agent` toolで `subagent_type: garmin-it-liaison` を呼び出す。プロンプトには (1)手順3のコーチのレビュー全文、(2)手順1の3章全文、(3)「コーチの指摘が実際のランナー（利用者）にとって妥当か確認し、優先度や懸念点を整理してください」という指示を渡す。戻り値として確認結果を得る。

5. **ログ記録**: `garmin/DESIGN_REVIEW_LOG.md` に日付見出しでエントリを追記する（ファイルが無ければ作成する）。フォーマットは下記「DESIGN_REVIEW_LOG.mdのエントリ形式」を参照。この書き込みは呼び出し元（このスキルを実行しているセッション自身）が行うこと。コーチ・情シスはファイル書き込み権限を持たないため代わりに書き込む。

6. **報告**: コーチのレビューと情シスの確認結果をユーザーに要約報告する。

## DESIGN_REVIEW_LOG.mdのエントリ形式

```markdown
## YYYY-MM-DD: 基本設計書レビュー

**コーチのレビュー**:
<手順3の出力>

**情シスの確認結果**:
<手順4の出力>
```

## 注意点

- コーチ・情シスの出力に技術的な内容（DBカラム名・APIパス・実装方法など）が混入していないか確認する。混入していれば役割逸脱として扱う
- `docs/04_DB設計.md`以降の技術章は絶対に読み込ませない・プロンプトに含めないこと
- 設計書の内容を変更してよいのはこのスキルではない（レビューのみ。修正はユーザーの判断で別途行う）
