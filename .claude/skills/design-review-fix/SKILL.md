# design-review-fix

`design-review`スキルによって`DESIGN_REVIEW_LOG.md`に記録済みのコーチ・情シスのレビュー結果を土台に、コンサル→コーディングの順で基本設計書(`docs/`配下)の修正プランを設計・適用する。コーチ・情シスの再レビューはこのスキルでは行わない。

引数（任意）: `DESIGN_REVIEW_LOG.md`内でどのレビューエントリを使うかの日付（`YYYY-MM-DD`）。省略時はファイル内で最も新しい日付見出しのエントリを使う。

## 手順

各ステップは前段の出力に依存するため、**Agent呼び出しは並列にせず必ず逐次実行**すること。

1. **レビュー結果読み込み**: `DESIGN_REVIEW_LOG.md`を`Read`する。
   - ファイルが存在しない、またはレビューエントリが1件も無い場合は、コーチ・情シスの呼び出しはこのスキル内では行わず、「先に`/design-review`を実行してください」とユーザーに伝えて終了する。
   - 存在する場合、引数で日付指定があればその`## YYYY-MM-DD:`見出しブロックを、無ければファイル内で最も新しい日付のブロック（同日複数あれば一番下のもの）を該当エントリとして抽出する（次の`##`見出し直前まで）。中身は「コーチのレビュー」「情シスの確認結果」の全文。

2. **設計書読み込み**: `docs/01_システム概要.md` `docs/02_機能一覧.md` `docs/03_画面設計.md`を`Read`する（`design-review`と同じ対象範囲。`04_DB設計.md`以降の技術章は読まない）。

3. **コンサル呼び出し**: `Agent` toolで`subagent_type: garmin-consultant`を呼び出す。プロンプトには (1) 手順1で抽出したレビュー全文（コーチ＋情シス）、(2) 手順2の設計書3章全文を渡し、「これは基本設計書に対するレビュー結果です。指摘を踏まえ、`docs/01〜03`のどのファイルのどの箇所をどう修正すべきか、具体的な修正プランを設計してください。技術章(DB/API設計等)への影響がある場合は言及してよいが、実装コードの詳細には踏み込まないこと」という指示を渡す。戻り値として修正プラン（対象ファイル／修正箇所／修正内容と理由）を得る。

4. **ログ記録(中間)**: `DESIGN_REVIEW_LOG.md`に新規エントリ（本日日付、下記「DESIGN_REVIEW_LOG.mdへの追記エントリ形式」）を追記する。この書き込みは呼び出し元（このスキルを実行しているセッション自身）が行うこと。コンサル・コーディングはファイル書き込み権限を持たない/持たせないため代わりに書き込む。

5. **コーディング呼び出し**: `Agent` toolで`subagent_type: garmin-coding`を呼び出す。プロンプトには手順3の修正プラン全文を渡し、「このプランに従って`docs/01_システム概要.md` `02_機能一覧.md` `03_画面設計.md`のうち該当するファイルを直接編集してください。プランに明記のない追加修正やアプリの実装コードの変更は行わないこと」と明示する。戻り値として変更したファイル一覧と修正サマリを得る。

6. **ログ追記・報告**: 手順4のエントリに「コーディングによる修正結果」を追記する。ユーザーには、元にしたレビュー日付・修正したファイル・修正内容の要約を報告する。

## DESIGN_REVIEW_LOG.mdへの追記エントリ形式

```markdown
## YYYY-MM-DD: 基本設計書 修正プラン（元レビュー: YYYY-MM-DD）

**元となったレビュー概要**:
<手順1で抽出したコーチ・情シスの内容>

**コンサルの修正プラン**:
<手順3の出力>

**コーディングによる修正結果**:
<手順5の出力>
```

既存の`design-review`によるレビューのみのエントリとは別に、修正パスとして日付付きの新規エントリを積み増す形にする（過去エントリは書き換えない）。

## 注意点

- コーチ・情シスはこのスキル内では呼び出さない（レビューのやり直しは`/design-review`の役割）。
- コーディング担当が編集してよいのは`docs/01_システム概要.md` `02_機能一覧.md` `03_画面設計.md`のみ。`04_DB設計.md`以降の技術章やアプリの実装コード（`backend/` `frontend/`）は対象外とし、コンサルの修正プランがそれらに触れる場合は「要別途検討」としてユーザーに報告するに留める。
- コンサルの出力に実装コードの詳細（関数名・具体的なコード片など）が混入していないか確認し、混入していれば役割逸脱として扱う。
- 修正後、コーチ・情シスによる承認確認は`/design-review-approve`で行う（このスキルでは行わない）。
