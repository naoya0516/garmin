# coach-consult

`feature-pipeline`の前半2段（コーチ→情シス）だけを、単一機能ではなく探索的・複数提案モードで回すスキル。`garmin-coach`/`garmin-it-liaison`自体は変更せず、呼び出し元プロンプトでタスクの枠組みを上書きする（`design-review`と同じパターン）。コンサル・コーディング・コードレビューは呼ばない（実装まで踏み込まない）。

引数（任意）: ランナーからの相談内容の原文。省略時は直前の会話から相談内容を特定してユーザーに確認する。

## 手順

各ステップは前段の出力に依存するため、**Agent呼び出しは並列にせず必ず逐次実行**すること。

1. **現状把握**: `garmin/PLAN.md`、`garmin/docs/02_機能一覧.md`（機能一覧および「対象外機能（将来検討）」欄）、`garmin/NEXT_PHASE_BACKLOG.md`を`Read`し、現状アプリが利用者に提供している機能一覧（利用者目線）と、既に将来検討として挙がっている候補を把握する（コーチが重複提案しないようにするため）。

2. **ナレッジ読み込み**: `garmin/.claude/knowledge/running-science/` 配下の全ファイルを`Read`する。

3. **コーチ呼び出し**: `Agent` toolで `subagent_type: garmin-coach` を呼び出す。プロンプトには (1) ランナーの相談内容原文、(2) 手順1の現状機能一覧、(3) 手順2のナレッジ全文、(4) 次のモード上書き指示を含める:
   > 「今回は通常の『1回1機能』ではなく探索的な相談です。3〜5件（目安4件）の機能候補を、いつもの4項目フォーマット（提案機能／ランニング科学的根拠／期待される効果／ランナーにとっての価値）を候補ごとに繰り返す形で提案してください。相談内容がカバーする複数の観点（例: 過去の実施内容を評価する／将来の練習を計画する）それぞれに最低1件は候補を含め、各候補がどの観点に効くかを明記してください。候補同士が同じ切り口の焼き直しにならないよう視点を変えてください。」

4. **情シス呼び出し**: `Agent` toolで `subagent_type: garmin-it-liaison` を呼び出す。プロンプトには (1) 手順3のコーチの提案全文（複数候補）、(2) 手順1の現状機能一覧、(3) 次のモード上書き指示を含める:
   > 「今回はコーチから複数（3〜5件）の候補が提案されています。1件にまとめず、候補ごとに確認結果・ユーザーストーリー・懸念点を簡潔にまとめてください。加えて、全候補を見渡した上で、システム観点（データの持ち方・既存機能との整合性・運用負荷）から見た着手しやすさの目安を候補ごとに一言添えてください。」

5. **ログ記録**: `garmin/COACH_CONSULT_LOG.md`に新規エントリ（本日日付、下記フォーマット）を追記する（ファイルが無ければ作成する）。この書き込みは呼び出し元（このスキルを実行しているセッション自身）が行うこと。コーチ・情シスはファイル書き込み権限を持たないため代わりに書き込む。

6. **バックログ追記**: `garmin/NEXT_PHASE_BACKLOG.md`の「検討リスト」表に、コーチが提案した候補ごとに1行（追加日／項目／概要1〜2文／関連=`COACH_CONSULT_LOG.md`の該当エントリへの参照）を追記する。

7. **報告**: ユーザーには、全候補・情シスの確認結果・優先度の目安を要約報告する。末尾に「気に入った候補があれば`/feature-pipeline <その候補の概要>`で個別に設計〜実装まで進められる」「基本設計書（`docs/`）への正式反映は実装が固まった後に別途明示的な指示のもとで行う」旨を案内する。

## COACH_CONSULT_LOG.mdへの追記エントリ形式

```markdown
## YYYY-MM-DD: <相談内容の要約>

**ランナーからの相談**:
<相談内容原文>

**コーチ提案（複数候補）**:
<手順3の出力>

**情シス確認結果（候補ごと＋優先度所見）**:
<手順4の出力>
```

## 注意点

- コーチ・情シスの出力に技術的な内容（ファイルパス、DBカラム名、API名など）が混入していないか確認し、混入していれば役割逸脱として扱う。
- コンサル・コーディング・コードレビューはこのスキルからは呼ばない（実装は行わない。実装したい候補が決まったら`feature-pipeline`を個別に使う）。
- `docs/`配下（基本設計書）は手順1で現状把握のために`Read`するのみで、絶対に編集しない（顧客合意済みのため、明示的な指示があるまで修正しない運用）。
- 過去のログエントリは書き換えない。常に新規エントリとして積み増す。
- 1回の相談で複数候補を扱う点が`feature-pipeline`（1回1機能）との違い。実装まで踏み込まない点が`design-review`/`issue-discuss`と共通。
