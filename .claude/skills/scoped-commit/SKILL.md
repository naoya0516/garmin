# scoped-commit

直前の作業でできたgitの変更を、意味のある単位に分割してコミットする。

引数（任意）: 今回の変更内容の要約（feature-pipelineやdesign-review-fixから渡される実装サマリなど）。省略時は`git status`/`git diff`から変更内容を自分で把握する。

## 手順

1. **変更確認**: `git status --porcelain=v1` で変更ファイル一覧を取得する（追跡外ファイルも含む）。変更が無ければ「コミットする変更はありません」と報告して終了する。

2. **差分把握**: `git diff`（未ステージ）と`git diff --cached`（ステージ済み）を確認し、新規ファイルは`Read`で中身を見て、各ファイルがどんな変更かを把握する。ファイル数が多い場合は`git diff --stat`で概観してから必要なファイルだけ詳細を見る。

3. **機微情報チェック**: 新規/変更ファイルに認証情報・トークン・個人情報など機微情報が含まれていないか確認する。`.gitignore`対象（例: `.claude/settings.local.json`）が万一`git status`に出てきた場合は、意図的な変更か確認してから扱う。

4. **グルーピング**: 変更ファイルを以下の観点でグループ化する。1グループ=1コミットとする。
   - 関心事が異なるものは分ける。例: 基本設計書(`docs/`配下)の修正 / アプリ実装コードの修正 / `.claude/`配下のエージェント・スキル定義の変更、は原則別コミット。
   - ただし「ある変更を行い、その結果を`FEATURE_LOG.md`や`DESIGN_REVIEW_LOG.md`に追記した」のように、ログ更新が特定の変更と1対1で対応し、分けると意味が通らなくなる場合は同じコミットにまとめる。
   - アプリ実装コードの中でも、無関係な複数機能を同一セッションで触った場合は機能ごとに分ける。一方、DB/API/UIの変更が1つの機能追加のために一体で必要なものはレイヤーで機械的に分割せず1コミットにまとめる。
   - タイポ修正や些末な1行修正のためだけに毎回コミットを刻まない。意味のある単位まで束ねる。
   - 目安: 1コミットは「それだけ見れば何をしたか・なぜかが説明できる」単位にする。ファイル数=コミット数にしない。多くのケースで数コミット程度に収まるはず。

5. **方針確認**: `git add`や`git commit`は一切実行せず、手順4で決めたグルーピング案をユーザーに提示して承認を求める。提示する内容は各グループごとに「コミットメッセージ案（`<category>: 要約`）」「含めるファイル一覧」。ユーザーから明示的な承認（OK/進めて等）が得られるまで次の手順に進まない。修正指示があれば（グループの分け方・カテゴリ・要約を）反映してから再度提示する。

6. **コミット実行**: 手順5で承認された案の通りに、グループごとに `git add <対象ファイルのみ>` → `git commit -m "..."` を行う。`git add -A`や`git add .`は使わず、グループに含まれるファイルを明示的に指定する。コミットメッセージは:
   - フォーマットは `<category>: <要約>` の1行のみ。要約部分は日本語で20〜30文字程度に収め、1行目全体で50文字を超えないようにする。「なぜ」より詳しい背景が要る場合でも本文は付けず、要約に収まらない詳細はユーザーへの報告で補う。
   - `<category>` は以下から最も当てはまるものを1つ選ぶ（迷ったら実質的な変更内容で判断する）:
     - `feat`: 新機能・新規実装の追加
     - `fix`: 不具合修正
     - `docs`: `docs/`配下の設計書の追加・修正
     - `refactor`: 挙動を変えないコード整理
     - `chore`: `.claude/`配下のエージェント・スキル定義や`PLAN.md`など運用系ファイルの変更
   - 例: `docs: 画面設計にペース分析画面を追加` / `feat: 週間走行距離グラフを実装` / `chore: scoped-commitスキルを追加`
   - 末尾（本文欄）に `Co-Authored-By: Claude Sonnet 5 <noreply@anthropic.com>` を付与する（要約行そのものには含めない）

7. **結果確認・報告**: 全グループのコミットが終わったら `git status` で作業ツリーがクリーンなことを確認し、`git log --oneline -n <作成したコミット数>` を添えてユーザーに報告する。何個のコミットに分けたか、それぞれの1行要約、pushはしていない旨を明記する。

## 注意点

- push は行わない。pushはユーザーの明示的な指示があった場合のみ別途行う。
- 既存コミットの`amend`や`reset`など破壊的操作は行わない。
- グルーピングに迷った場合は、過剰な分割よりも多少大きめのまとまりに倒す（分割しすぎて履歴が読みにくくなる方が害が大きい）。
- コミット対象にすべきか判断できないファイル（生成物、ローカル設定など）があれば、コミットする前にユーザーに確認する。
