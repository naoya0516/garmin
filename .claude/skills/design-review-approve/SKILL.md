# design-review-approve

`design-review-fix`によって`DESIGN_REVIEW_LOG.md`に記録済みの「修正プラン」エントリを対象に、そこで解消を試みた懸念点が実際の修正後の設計書で解消されているかをコーチ・情シスに確認させ、承認/要修正の判定を得る軽量な確認ステップ。新規の懸念点の提起や機能提案は求めない（それは`/design-review`の役割）。

引数（任意）: `DESIGN_REVIEW_LOG.md`内でどの「修正プラン」エントリを対象にするかの日付（`YYYY-MM-DD`、`## YYYY-MM-DD: 基本設計書 修正プラン`見出しの日付）。省略時はファイル内で最も新しい「修正プラン」エントリを対象にする。

## 手順

各ステップは前段の出力に依存するため、**Agent呼び出しは並列にせず必ず逐次実行**すること。

1. **対象エントリ読み込み**: `DESIGN_REVIEW_LOG.md`を`Read`する。
   - 「修正プラン」エントリ（`## YYYY-MM-DD: 基本設計書 修正プラン（元レビュー: YYYY-MM-DD）`見出し）が1件も無い場合は、コーチ・情シスの呼び出しは行わず、「先に`/design-review-fix`を実行してください」とユーザーに伝えて終了する。
   - 引数で日付指定があればその見出しのブロックを、無ければファイル内で最も新しい「修正プラン」見出しのブロック（同日複数あれば一番下）を対象エントリとして抽出する（次の`##`見出し直前まで）。中身は「元となったレビュー概要」「コンサルの修正プラン」「コーディングによる修正結果」の全文。
   - そのエントリの「コーディングによる修正結果」が「（後続ステップで追記）」等の未実施状態のままの場合は、まだ修正が完了していないとみなし、「`/design-review-fix`の修正がまだ完了していないようです」とユーザーに伝えて終了する。
   - 対象エントリより後に、同じ対象修正日付を指す「修正承認確認」エントリが既に存在する場合は、そのままユーザーにその旨（既に承認確認済みであること、既存の判定内容）を伝えて終了してよい（再確認したい場合はユーザーに再実行の意思を確認する）。

2. **設計書読み込み(修正後)**: `docs/01_システム概要.md` `docs/02_機能一覧.md` `docs/03_画面設計.md`を`Read`する（修正が適用された後の現在の内容。`04_DB設計.md`以降の技術章は読まない）。

3. **コーチによる承認確認**: `Agent` toolで`subagent_type: garmin-coach`を呼び出す。プロンプトには (1) 手順1の「元となったレビュー概要」のうちコーチの懸念点部分、(2) 手順1の「コンサルの修正プラン」「コーディングによる修正結果」、(3) 手順2の設計書3章全文（修正後）を渡し、「これはあなたが以前指摘した懸念点に対する修正結果です。新しい懸念点の指摘や追加機能の提案は不要です。以前の懸念点それぞれについて、今回の修正で解消されたとみなせるか（解消／一部解消／未解消、理由付き）を判定し、全体として承認できるか（承認／要修正）を答えてください」という指示を渡す。戻り値として懸念点ごとの判定と総合判定を得る。

4. **情シスによる承認確認**: `Agent` toolで`subagent_type: garmin-it-liaison`を呼び出す。プロンプトには (1) 手順1の「元となったレビュー概要」のうち情シスの確認結果部分（要件・懸念点・曖昧な点）、(2) 手順1の「コンサルの修正プラン」「コーディングによる修正結果」、(3) 手順2の設計書3章全文（修正後）、(4) 手順3のコーチの承認確認結果を渡し、「以前あなたが整理した懸念点・曖昧な点が、今回の修正で利用者視点から見て妥当に解消されているかを確認してください。新しい要件の追加提案は不要です。懸念点・曖昧な点ごとの判定と、全体としての承認可否（承認／要修正）を答えてください」という指示を渡す。戻り値として判定を得る。

5. **ログ記録・報告**: `DESIGN_REVIEW_LOG.md`に新規エントリ（本日日付、下記フォーマット）を追記する。この書き込みは呼び出し元（このスキルを実行しているセッション自身）が行うこと。コーチ・情シスはファイル書き込み権限を持たないため代わりに書き込む。ユーザーには、対象にした修正プランの日付・総合判定（承認／要修正）・要修正の場合は残っている懸念点を要約報告する。要修正の場合は、`/design-review-fix`を再実行して追加修正するよう促す。

## DESIGN_REVIEW_LOG.mdへの追記エントリ形式

```markdown
## YYYY-MM-DD: 基本設計書 修正承認確認（対象修正: YYYY-MM-DD）

**コーチの承認確認**:
<手順3の出力>

**情シスの承認確認**:
<手順4の出力>

**総合判定**: 承認 / 要修正
```

## 注意点

- このスキルは「以前の懸念点が解消されたか」の確認に徹する。コーチ・情シスに新規の懸念点提起や機能提案をさせない（プロンプトで明示的に禁止する）。仮に新規の懸念点が出力に混入した場合は、この確認ステップの判定（承認／要修正）には含めず、ユーザーへの報告時に「今回の承認確認の対象外。別途`/design-review`で扱うべき新規指摘」として区別して伝える。
- コーチ・情シスの出力に技術的な内容（DBカラム名・APIパス・実装方法など）が混入していないか確認する。混入していれば役割逸脱として扱う。
- `docs/04_DB設計.md`以降の技術章は読み込ませない・プロンプトに含めないこと。
- 「要修正」判定の場合でも、このスキル自身は設計書の再修正を行わない（それは`/design-review-fix`の役割）。
- 過去のログエントリは書き換えない。承認確認は常に新規エントリとして積み増す。
