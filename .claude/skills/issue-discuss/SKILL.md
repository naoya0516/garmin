# issue-discuss

実装・設計を進める中で見つかった課題（例: 設計書間の矛盾、記載漏れ、判断の割れ）について、関係するエージェントに議論させ、最終的な結論を`ISSUE_LOG.md`に記録する。ユーザーへの一問一答ではなく、エージェント同士の意見のやり取りを経て結論を出す点が`design-review`系スキルとの違い。

引数（任意）: 課題の内容（何が問題か）。省略時は直前の会話・作業から課題を自分で特定してユーザーに確認する。

## 手順

各ステップは前段の出力に依存するため、**Agent呼び出しは並列にせず必ず逐次実行**すること。

1. **課題の整理**: 課題の内容を明確化する。関連する`docs/`配下のファイル、`PLAN.md`、既存の`DESIGN_REVIEW_LOG.md`/`FEATURE_LOG.md`/`ISSUE_LOG.md`を`Read`し、背景・経緯を把握する。

2. **参加エージェントの決定**: 課題の性質に応じて議論させるエージェントの組み合わせを決める。
   - 設計・実装レベルの技術的な論点（DB/API設計、実装方針など） → `garmin-consultant`（初期判断）→ `garmin-coder`（応答）
   - 利用者視点・機能要件レベルの論点 → `garmin-coach`または`garmin-it-liaison`を交える
   - 実装済みコードの品質に関わる論点 → `garmin-code-reviewer`を交える
   - 迷う場合はユーザーに確認する
   - 通常は2エージェントの往復（初期判断→応答）で足りる。応答側が明確に異論を述べ結論が割れた場合のみ、1人目に差し戻して3ラウンド目を行う

3. **1人目のエージェント呼び出し**: `Agent` toolで1人目を呼び出す。課題の背景・関連ドキュメントの内容を渡し、結論とその根拠を明確にするよう求める。

4. **2人目のエージェント呼び出し**: `Agent` toolで2人目を呼び出す。課題の背景に加えて手順3の判断内容を渡し、同意する点・異論がある点・最終的な結論を明確にするよう求める。

5. **（必要な場合のみ）差し戻し**: 手順4で明確な異論が出て結論が割れた場合、手順3のエージェントに手順4の応答を渡し最終見解を求める。

6. **ログ記録**: `ISSUE_LOG.md`に新規エントリ（本日日付、下記フォーマット）を追記する。ファイルが無ければ作成する。この書き込みは呼び出し元（このスキルを実行しているセッション自身）が行うこと。議論に参加したエージェントはファイル書き込み権限を持たない/持たせないため代わりに書き込む。

7. **報告**: ユーザーには、課題の内容・各エージェントの結論・最終的な結論と対応方針を要約報告する。実際の修正（ファイル編集）が必要な場合は、この時点では行わずユーザーに実施可否を確認する（このスキルは議論と記録に徹し、修正の実行は別途明示的な指示を受けてから行う）。

## ISSUE_LOG.mdへの追記エントリ形式

```markdown
## YYYY-MM-DD: <課題タイトル>

**発生した問題**:
...

**議論**:
<1人目のエージェントの判断>

<2人目のエージェントの応答>

（意見が割れた場合は追加ラウンドをここに記載）

**結論・対応**:
...

**関連ファイル/参照**:
...
```

## 注意点

- このスキル自体はファイルの実装修正を行わない（議論と記録のみ）。結論に基づく実際の修正は、ユーザーの承認を得てから該当エージェント（通常はコーディング）に別途依頼する。
- 議論の対象がコーチ・情シスの権限方針（コード非可視）に関わる場合、コード詳細をプロンプトに含めない。
- 過去のログエントリは書き換えない。常に新規エントリとして積み増す。
- 明確な結論に至らない（意見が割れたままユーザー判断が必要な）場合は、その旨を「結論・対応」に明記し、ユーザーに判断を委ねる。
