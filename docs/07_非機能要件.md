# 07. 非機能要件

個人利用のホビープロジェクトであることを前提に、過剰な要件は設定せず、実態に見合った現実的な水準を定める。

## 性能

- 想定利用者は1名（開発者本人）、同時アクセスなし
- `GET /api/activities`はDBからの読み取りのみのため高速。`POST /api/sync`はGarmin Connectとの通信を伴うため数秒〜十数秒かかることを許容する
- 厳密な性能目標（レスポンスタイムSLA等）は設定しない

## セキュリティ

- Garmin Connectの認証情報（`GARMIN_EMAIL` / `GARMIN_PASSWORD`）は`.env`ファイルに保存し、`.gitignore`で除外する
- トークンキャッシュ（`garth`が生成）も同様にリポジトリに含めない
- バックエンドAPIは`localhost`のみでリッスンし、インターネットには公開しない
- CORSは開発用フロントエンドのオリジン（`http://localhost:5173`）のみ許可する
- ユーザー認証・ログイン画面は設けない（本人のみが利用するローカル環境が前提のため）

## 可用性

- 常時稼働は不要。開発者が使いたいときにローカルでサーバーを起動する運用を前提とする
- 障害時の自動復旧・冗長化は考慮しない

## 運用・保守

- ログはMVPでは標準出力への出力のみとし、ログ収集基盤等は導入しない
- DBスキーマ変更が必要になった場合、MVPの間は`Base.metadata.create_all()`による素朴な運用とし、Alembic等のマイグレーションツールはスキーマが安定してから導入を検討する
- 非公式のGarmin連携ライブラリ（[06_外部インターフェース設計](06_外部インターフェース設計.md)）に依存しているため、Garmin側の仕様変更時は同期機能が動作しなくなるリスクがある。定期的な動作確認を推奨する
- バックアップ: SQLiteファイル（`data/garmin.db`）1つで完結するため、必要であれば手動コピーでバックアップ可能
