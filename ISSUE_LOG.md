# 課題管理ログ

開発を進める中でエージェント間・設計書間で発生した論点（矛盾・記載漏れ・判断の割れなど）について、何が問題になったか・どう議論したか・どう解決したかを記録する。`issue-discuss` スキルを実行するたびに、以下の形式で1エントリを追記する（過去エントリは書き換えない）。

```markdown
## YYYY-MM-DD: <課題タイトル>

**発生した問題**:
...

**議論**:
<1人目のエージェントの判断>

<2人目のエージェントの応答>

（意見が割れた場合は追加ラウンドをここに記載）

**結論・対応**:
...

**関連ファイル/参照**:
...
```

## 2026-07-17: F-03用APIエンドポイントが05_API設計.mdに未反映

**発生した問題**:

`docs/08_詳細設計.md`（実装前の詳細設計、gitタグ`v1.1.0-design`）を作成する過程で、F-03（走行サマリー表示）を実現するためのバックエンドAPIとして`GET /api/activities/summary`という新規エンドポイントを設計した。しかし確定済みの基本設計書`docs/05_API設計.md`（gitタグ`v1.0.0-design`）のエンドポイント一覧表には`GET /api/activities`と`POST /api/sync`のみが記載されており、F-03に対応するエンドポイントの記載が無かった。担当コーディングエージェントは「01〜07は変更しない」というタスク指示に従い、08側に「暫定案・要確認」と注記するに留めていたが、ユーザーからこの経緯について「なぜ基本設計と異なる実装判断をしたか」の説明を求められたことで、05を放置してよいか（＝基本設計書の修正が必要か）という論点が浮上した。

**議論**:

コンサル（garmin-consultant）の判断:
「修正すべき。ただし軽微な追記で足りる」。`01_システム概要.md`はF-03を今回のMVP対象と明記し、`02_機能一覧.md`はF-03を承認済み機能一覧に正式掲載、`03_画面設計.md`の操作フローシーケンス図は「サマリー取得(F-03)」というAPI呼び出しの発生自体を明記している。01〜03（承認済み）はF-03のAPI呼び出しを前提にしているのに契約を定める05に該当行が無いのは内部矛盾であり、「05は技術章でコーチ・情シスのレビュー対象外」なのは内容審査プロセスの範囲の話であって01〜03との整合を免除する理由にはならない。`DESIGN_REVIEW_LOG.md`を確認すると、前任コンサルが`design-review-fix`当時に既に「APIはサマリー専用の新規エンドポイント追加を推奨」と言及していたが、当時のスコープが01〜03に限定されていたため05への反映が先送りされていた経緯がある。修正内容は08が既に提示した`GET /api/activities/summary`・`ActivitySummaryOut`を転記する程度で完結し、新規のUX判断を伴わないためコーチ・情シスの再レビューは不要。パス名の技術判断（`/api/activities/summary` vs `/api/summary`）もコンサル・コーディング間の合意で足りる。

コーディング（garmin-coding）の応答:
コンサルの4点すべてに同意。実装者の立場からも「08を書く際に実際にこのギャップに直面し、独自にエンドポイントを設計する羽目になった」という実務上の裏付けを追加。前任コンサルの推奨が先送りされていた経緯も`DESIGN_REVIEW_LOG.md`で事実確認し同意。加えて2点補足: (1) 05は`v1.0.0-design`タグで確定済みのため、タグ後に追記する場合は「いつ・なぜ追記したか」が分かる記録を残すべき、(2) 実際に修正する場合はコーディング自身が転記作業を担当する進め方を提案。結論として「今回のタイミングで05_API設計.mdを修正すべき」に同意。

**結論・対応**:

ユーザー承認のもと、`docs/05_API設計.md`に`GET /api/activities/summary`のエンドポイントを追記する（コーチ・情シスの再レビューは行わない。理由は上記議論の通り、01〜03で承認済みの仕様を技術文書に転記するのみのため）。あわせて`docs/08_詳細設計.md`の「暫定案・要確認」の注記を、05へ反映済みである旨に更新する。担当はコーディング（garmin-coding）。作業内容は別途コミットログ・本エントリで追跡可能とする。
