## 2026-07-17: 基本設計書レビュー

**コーチのレビュー**:

注記: 参照予定だったランニング科学ナレッジフォルダ（garmin/.claude/knowledge/running-science/）が空だったため、トレーニングコーチとしての一般的な知見に基づいてレビューしています。

### 良い点

- 走行データを一元管理し、将来の振り返り・分析につなげるという方向性自体は、トレーニング管理の基本として正しいアプローチです。継続的な記録なしにトレーニングの質を改善することはできません。
- 生データをそのまま保持しておく設計方針(raw_jsonの保持)は、後から新しい分析軸を追加できる余地を残しており、将来的な指標追加(心拍ゾーン分析やトレーニング負荷計算など)を阻害しない点は評価できます。
- 距離・時間・ペース・平均心拍・カロリーという基本指標を収集対象にしている点は、トレーニングの土台となる最低限のデータとして妥当です。

### 懸念点

1. **「量」は見えるが「強度・負荷」が見えない** — 1件ごとの表示だけでは、オーバートレーニングや怪我のリスクにつながる「負荷の急増」に本人が気づきにくい。
2. **平均心拍だけでは強度の中身が分からない** — ジョグとインターバルの区別ができず、心拍ゾーン滞在時間の情報が本来必要。
3. **個人の基準値(最大心拍数・閾値心拍数など)を持つ仕組みがない** — 将来のゾーン分析やペース戦略提案の前提となる基準がない。
4. **ウォーキングとランニングが同列に並ぶ** — 将来の集計・分析で種別ごとの扱いを分ける前提を早めに意識しないと、負荷計算のロジックがねじれる可能性。
5. **リカバリー(回復)の視点が完全に欠落している** — 安静時心拍・睡眠・主観的疲労感などを扱う余地が現状ない。DB設計段階で後付け可能性を意識すべき。

### 追加で検討すべき機能(優先度順)

1. 週間・月間の走行距離/時間サマリー
2. 個人の基準心拍(最大心拍数・安静時心拍数など)の登録・保持
3. トレーニング負荷トレンドの簡易表示
4. アクティビティ種別ごとのフィルタ・集計の分離

**情シスの確認結果**:

**総評: 要調整（MVPスコープの逸脱を含むため、次フェーズとして切り出すことを推奨）**

コーチの指摘4点はいずれもランナーにとって実際に価値のある視点だが、現行の基本設計書では「取得・保存・一覧表示」のみがMVPスコープと明記されており、分析機能・フィルタ機能は明確に対象外。ランナー視点で優先度をつけ直すと以下の通り。

| コーチの提案 | ランナー視点での妥当性 | MVPとの整合性 |
|---|---|---|
| 1. 週間・月間サマリー表示 | 妥当性高い。既存の一覧表示（F-02）の自然な延長で、ランナーの操作は増えない | 次フェーズ最優先候補 |
| 2. 個人の基準心拍の登録・保持 | 将来必要だが、新規入力画面・入力の手間が発生し「今すぐ使いたい機能」ではない | 今回は見送り、DBスキーマ設計時に拡張余地だけ確保 |
| 3. トレーニング負荷トレンド簡易表示 | 有用だが算出ロジックの妥当性検証等、検討事項が多い | 分析機能そのもので明確に対象外 |
| 4. 種別ごとのフィルタ・集計分離 | ユーザーから見える機能ではなく内部データ設計の話 | 設計方針の申し送り事項として扱う |

既存機能（F-02）との重複はなし。

**要件（ユーザーストーリー形式）**:
ランナー（利用者）として、アクティビティ一覧画面で直近1週間・直近4週間の合計走行距離と合計時間のサマリーを確認したい。なぜなら、1件ずつの記録を見るだけでは負荷の変化に気づきにくく、怪我やオーバートレーニングを防ぐために練習量を俯瞰的に把握しておきたいから。

**懸念点・曖昧な点**:
- 集計期間の起点（週の始まりが月曜か日曜か、暦週か直近7日間か）が未定義
- F-01の同期範囲（直近30日）と4週間サマリーの整合性、過去同期済みデータの扱い
- サマリーにウォーキングを含めるか、ランニングのみを対象にするか未決定
- 提案2（基準心拍登録）・3（負荷トレンド表示）は今回の要件に含めず、将来機能として別途切り出すことを推奨
