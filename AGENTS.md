# 機能要求パイプライン エージェント仕様

Garminアプリに新機能を育てていくための4エージェント構成。定義ファイルは `.claude/agents/` と `.claude/skills/feature-pipeline/SKILL.md` にある。

## エージェント一覧

| エージェント | ファイル | 役割 | ツール権限 |
|---|---|---|---|
| コーチ | `.claude/agents/garmin-coach.md` | ランニング科学の知見から機能を提案 | WebSearch, WebFetch（コード不可視） |
| 情シス | `.claude/agents/garmin-it-liaison.md` | 利用者視点で提案を確認し要件に整形 | なし（コード不可視） |
| コンサル | `.claude/agents/garmin-consultant.md` | 要件を受けて設計方針を検討 | Read, Grep, Glob（読み取りのみ） |
| コーディング | `.claude/agents/garmin-coder.md` | 設計に基づき実装 | 全権限（唯一の編集担当） |

ランナー（アプリ利用者本人）はエージェント化しない。

## 各エージェントの詳細

### コーチ（garmin-coach）
- **入力**: ①ランナーの相談内容（任意、空なら「次に追加すべき機能を一般的な観点から提案して」扱い）、②現在のアプリの利用者向け機能一覧、③ `.claude/knowledge/running-science/` のランニング科学ナレッジ全文（スキルが読み込んでプロンプトに注入。コーチ自身はReadツールを持たないため、コード非可視の境界は保ったまま知識だけを受け取る）
- **出力**: 提案機能 / ランニング科学的根拠 / 期待される効果 / ランナーにとっての価値（技術用語は使わない）
- 通常はfeature-pipelineスキル経由で呼び出す

### 情シス（garmin-it-liaison）
- **入力**: コーチの提案全文＋利用者向け機能一覧
- **出力**: 確認結果（妥当/要調整/重複の可能性）/ ユーザーストーリー形式の要件1件 / 懸念点
- ツールを一切持たないため、渡されたテキストのみで判断する

### コンサル（garmin-consultant）
- **入力**: 情シスの要件全文
- **動作**: `garmin/PLAN.md`や既存コード（実装が進めば`backend/`/`frontend/`）を`Read`/`Grep`/`Glob`で自ら確認
- **出力**: 設計概要 / 影響範囲（DB・API・UI）/ 既存アーキテクチャとの整合性 / 実装時の注意点
- ファイルの作成・編集は一切行わない

### コーディング（garmin-coder）
- **入力**: コンサルの設計方針全文
- **動作**: `PLAN.md`の規約に従って実装。過剰な機能追加やリファクタリングはしない
- **出力**: 実装内容の要約 / 変更ファイル一覧 / 動作確認結果（実際のGarminアカウント確認が必要な場合はその旨明記）

## 利用方法

1. Garminディレクトリを開いた状態で `/feature-pipeline` を実行（引数任意、例: `/feature-pipeline 最近テンポ走がつらい`）
2. スキルが以下を**逐次**実行する:
   - `PLAN.md`/`FEATURE_LOG.md`から現状機能を要約
   - `.claude/knowledge/running-science/`のランニング科学ナレッジを読み込む
   - コーチ→情シス→コンサルの順にAgent toolで呼び出し、出力を次段へ引き継ぐ
   - コンサルの設計結果までを`FEATURE_LOG.md`に追記（この書き込みはスキル実行元本体が行う。コーチ/情シス/コンサルはファイル書き込み不可のため）
   - コーディングを呼び出して実装させ、結果を同エントリに追記
3. 実行結果は `FEATURE_LOG.md` に日付見出し付きで蓄積される

現状はアプリ本体が未実装（`PLAN.md`の設計のみ）のため、まずはMVPを実装してからこのパイプラインを回すのがおすすめ。

## design-reviewスキル（基本設計書のレビュー）

`garmin/docs/` の基本設計書をコーチ・情シスにレビューさせるための別スキル。定義ファイルは `.claude/skills/design-review/SKILL.md`。

- 対象はコーチ・情シスの権限方針（コード非可視）と整合する `docs/01_システム概要.md` `02_機能一覧.md` `03_画面設計.md` の3章のみ。`04_DB設計.md`以降の技術章は読ませない
- 実行方法: `/design-review`（引数なし）
- 流れ: 3章＋ランニング科学ナレッジをコーチに読ませてレビュー→情シスが利用者視点で確認→`DESIGN_REVIEW_LOG.md`に記録しつつユーザーへ報告（コンサル・コーディングは呼ばない）
- 設計書を更新するたびに何度でも実行できる

## issue-discussスキル（課題管理・エージェント間の議論）

開発中に見つかった論点（設計書間の矛盾、記載漏れ、判断の割れなど）を関係エージェント同士に議論させ、結論を`ISSUE_LOG.md`に記録するための別スキル。定義ファイルは `.claude/skills/issue-discuss/SKILL.md`。

- 課題の性質に応じて議論させるエージェントを選ぶ（技術的な論点はコンサル→コーディング、利用者視点の論点はコーチ・情シスを交える等）
- 実行方法: `/issue-discuss <課題の内容>`（引数省略可）
- 議論と記録のみを行い、結論に基づく実際の修正は行わない（ユーザー承認後に別途実施）
- 何か論点にぶつかるたびに何度でも実行できる。過去のエントリは書き換えず積み増す
